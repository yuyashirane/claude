# proposal-generator v2.0

## 概要

あしたの会計事務所の税務顧問サービス提案書を、議事録（音声文字起こし含む）から自動生成するスキル。
v1の1次元料金テーブル（売上+仕訳数の一体型）を、v2では売上クラス×仕訳数ティアの2次元マトリクスに拡張し、
記帳代行・自計化・相談サポートをモジュール単位で組み合わせ可能にした。
出力モードは「お任せ型（single）」と「4択型（multi）」の2種類。
売上1億円超の案件は自動判定せず、スタッフが料金を手動設定できる仕組みを備える。

## ワークフロー

### Step 1: 入力資料の収集

議事録のパス、お客様名、面談日を確認する。

**標準パス規則:**
```
G:\共有ドライブ\06_見込み客\{YYYYMMDD}_{お客様名}\
```

**確認事項:**
- 議事録ファイル（.txt, .md, .docx, .pdf 等）の場所
- 複数ファイルがある場合は全て読み込む（面談メモ、事業計画書、名刺情報等）
- ユーザーがローカルパスを指定した場合はそのパスを使用

**議事録が音声文字起こしの場合の注意:**
- 話者の区別がない場合がある → 文脈から「顧客発言」と「事務所側発言」を推定
- 固有名詞の誤認識が多い（例: 「freee」→「フリー」「ふりー」、「インボイス」→「インボイズ」）
- 数値の読み違い（「3000万」→「3,000万」「三千万」表記揺れ）
- 「えーと」「あの」等のフィラーは無視

### Step 2: 情報抽出

議事録から以下の情報を抽出する。

#### 必須項目（config生成に不可欠）

| 項目 | config パス | 抽出のコツ |
|------|------------|-----------|
| 会社名（法人格含む） | `client.name` | 「株式会社」「合同会社」等の法人格を含める。議事録冒頭・名刺情報に記載されることが多い |
| 業種・事業内容 | `client.industry`, `client.businessDescription` | 「何をやっている会社か」を端的に。複合事業の場合は全て記載 |
| 売上規模（年商） | `client.annualRevenue` | 「年商」「売上」「年間売上」等のキーワード周辺。「億」「万」単位に注意 |
| 月間仕訳数（おおよそ） | `client.monthlyTransactions` | 明示されない場合が多い。「取引先何社」「毎月の経費件数」等から推定ヒントを探す |
| 決算月 | `client.fiscalMonth` | 「何月決算」「期末は」等。記載なければ省略（デフォルト3月と仮定しない） |
| 経理体制の希望 | `pricing.selectedModules` | 「自分で入力」「freee使ってる」→ 自計化、「全部お願いしたい」「丸投げ」→ 記帳代行 |
| 紹介経由かどうか | — | 「〜さんのご紹介」「〜からの紹介」→ 紹介値引き候補 |

#### 推奨項目（あれば記載）

| 項目 | config パス | 備考 |
|------|------------|------|
| 設立日 | `client.established` | YYYY-MM-DD形式。「設立予定」の場合はその旨をnoteに |
| 資本金 | — | config v2.0にフィールドなし。challengesの参考情報として使用 |
| 従業員数 | `client.employees` | 年末調整の人数算定に必要 |
| 代表者名 | — | config v2.0にフィールドなし。挨拶文のカスタマイズに使用可能 |
| 所在地 | — | 訪問対応の記載に影響 |
| 顧客の予算感 | — | 「月XX万くらいを想定」→ 値引き調整の参考 |
| 消費税の課税区分 | `pricing.consumptionTaxMethod` | 「簡易課税」「原則課税」「免税」等 |
| 課題・要望（自由記述） | `challenges` | 議事録全体から抽出 |

#### 抽出できなかった場合の対処

**原則: 推測しない。「不明」と記録し、ユーザーに確認を求める。**

```
⚠️ 以下の項目が議事録から確認できませんでした:
- 月間仕訳数: 取引構造から推定すると50〜100件程度と思われますが、確認が必要です
- 消費税の課税区分: 売上規模から課税事業者と推定されますが、簡易/原則の選択は不明です

→ これらの項目を確認いただけますか？仮置きで進める場合はその旨お知らせください。
```

必須項目のうち **会社名** と **売上規模** が不明の場合は config 生成を中断し、確認を求める。
それ以外の必須項目は仮置き値で進め、config内にコメント相当の注記を残す。

### Step 3: outputMode の判定

議事録の内容から出力モードを判定する。以下の優先順位で判定:

```
判定1: 議事録に「複数プラン」「比較したい」「選びたい」等の発言あり
        → proposal_multi

判定2: 議事録に「記帳代行で」「自計化で」「お任せで」と明確な希望あり
        → proposal_single

判定3: 紹介経由案件（紹介者の名前が出てくる）
        → proposal_multi（紹介者向けに選択肢を提示する慣習）

判定4: 上記いずれにも該当しない
        → proposal_single（デフォルト）
```

**判定理由は必ず `meta.outputModeReason` に日本語で記録する。**

例:
- `"顧客が記帳代行を明確に希望、サービス構成は確定済みのため単一プラン提案"`
- `"紹介経由案件で顧客が記帳代行と自計化を比較検討中のため、4プラン併記で提案"`

**サンプル参照:**
- proposal_single の典型例: `samples/config-v2-megasolar.json`, `samples/config-v2-bangkok-solar.json`
- proposal_multi の典型例: `samples/config-v2-backup-multi.json`

### Step 4: 売上クラス判定

`client.annualRevenue` に基づき売上クラスを判定する。

```
annualRevenue <  10,000,000  → A   （売上1千万円未満）
annualRevenue <  30,000,000  → B   （売上3千万円未満）
annualRevenue <  50,000,000  → C   （売上5千万円未満）
annualRevenue < 100,000,000  → D   （売上1億円未満）
annualRevenue >= 100,000,000 → OVER（売上1億円以上）
```

#### OVERクラスの処理

売上1億円以上は料金表の範囲外。Claudeが機械的に料金を決定してはならない。

**必須設定:**
- `client.salesClass`: `"OVER"`
- `client.requiresManualPricing`: `true`
- `client.manualPricingReason`: 理由を記載（例: `"売上2億円のため料金表対象外、個別見積り"`）
- `pricing.manualPricing.enabled`: `true`
- `pricing.manualPricing.reason`: 上記と同内容

**テンプレ値のガイドライン（スタッフ調整前の初期値）:**

| 売上規模 | 月次基本料（記帳代行） | 月次基本料（自計化） | 法人税申告 | 消費税申告 |
|---------|---------------------|-------------------|-----------|-----------|
| 1〜2億円 | 50,000〜70,000円 | 30,000〜45,000円 | 200,000〜250,000円 | 60,000〜80,000円 |
| 2〜3億円 | 65,000〜85,000円 | 40,000〜55,000円 | 220,000〜280,000円 | 70,000〜90,000円 |
| 3億円超 | 80,000〜120,000円 | 50,000〜70,000円 | 250,000〜350,000円 | 80,000〜120,000円 |

※ あくまでスタッフ判断の叩き台。仕訳数が少ない場合（粗利ベースで判定可能な場合）は下限寄りに設定。

**サンプル参照:**
- 売上1.4億・仕訳少量: `samples/config-v2-bangkok-solar.json`（月次70,000→出精-15,000で55,000）
- 売上2億・粗利ベース判定: `samples/config-v2-kanuu-shoun.json`（粗利1千万でAクラス水準、月次35,000）
- 売上2億・複合事業: `samples/config-v2-backup-multi.json`（月次65,000、4プラン併記）

### Step 5: 値引きの判定

値引きは2種類。いずれも最終金額はスタッフが調整する前提で、Claudeはテンプレ値を提案する。

#### ご紹介値引き（introduction discount）

**判定トリガー:**
- 議事録に紹介者の名前が出てくる
- 「〜さんからのご紹介」「〜先生にご紹介いただいた」等の発言

**格納先:** `pricing.manualDiscounts.introduction`（single）/ 各plan内の`manualDiscounts.introduction`（multi）

**金額の目安:**
- 月次: -3,000〜-12,000円（売上規模・プランによる）
- 年次: -10,000〜-30,000円

#### 出精値引き（volume discount）

**判定トリガー:**
- 「予算が厳しい」「他社と比較中」「もう少し安くならないか」等の発言
- 取引量が料金クラスに対して明らかに少ない（例: Dクラスだが月30仕訳）
- 休眠会社・事業未開始で取引がほぼない

**格納先:** `pricing.manualDiscounts.volume`

**金額の目安:** 売上クラスの基本料金の10〜30%程度

**サンプル参照:**
- 仕訳少量による出精: `samples/config-v2-megasolar.json`（月-5,000、年-50,000）
- 事業未開始の大幅値引き: `samples/config-v2-nihon-ample.json`（月-5,000、年-40,000）
- OVERクラスでの出精: `samples/config-v2-bangkok-solar.json`（月-15,000、年-80,000）

**注意:** 値引き理由は `monthlyReason` / `annualReason` に必ず記載する。

### Step 6: 課題とサービスの紐付け

議事録から顧客の課題を抽出し、それぞれに対応するサービスを紐付ける。

#### 課題抽出のポイント

- 顧客が明示的に述べた悩み・要望を最優先
- 業種・事業構造から推定される一般的課題も追加可能（ただし議事録に根拠があること）
- 課題数は3〜6個が目安。多すぎると焦点がぼやける

#### 紐付けルール

各 `challenge` の `linkedServiceIds` に、その課題を解決するサービスの `id` を配列で指定する。

```json
{
  "id": "ch01",
  "title": "外貨建取引の会計処理",
  "detail": "売上・仕入の99%以上がドル建てであり...",
  "linkedServiceIds": ["sv01", "sv02"]
}
```

#### 業種別の典型課題パターン

**太陽光発電事業:**
- 固定資産の減価償却管理
- 償却資産申告
- 将来の売却を見据えた税務対応
- → サンプル: `config-v2-megasolar.json`, `config-v2-bangkok-solar.json`

**海運代理業（外貨取引中心）:**
- 外貨建取引の会計処理
- 消費税の還付対応
- 設立届出
- → サンプル: `config-v2-kanuu-shoun.json`

**飲食業（複合事業）:**
- 軽減税率対応
- 現金管理の強化
- 法人成りに伴う経理体制構築
- → サンプル: `config-v2-backup-multi.json`

**休眠・事業未開始会社:**
- 休眠状態の会社維持管理
- 将来の事業開始への備え
- → サンプル: `config-v2-nihon-ample.json`

### Step 7: config.json の生成

抽出・判定した情報をもとに config.json v2.0 を生成する。

**出力先:** `references/pricing/config_{お客様名}.json`
（お客様名は半角英数またはローマ字。日本語ファイル名は避ける）

#### カラーテーマの選択

config.json の `meta.theme` でカラーテーマを指定可能:
- `blue`: スタンダード、テック系・知的な印象（デフォルト）
- `orange`: 親しみやすさ・温かみ、中小企業向け（あしたの会計事務所 旧ブランド）
- `gray`: フォーマル、上場企業・大規模法人向け、控えめで品格

議事録から会社の雰囲気を読み取り、適切なテーマを推奨する:
- 中小企業・スタートアップ・サービス業 → orange
- IT・テック・新興企業 → blue
- 大手・上場準備・士業・金融 → gray

判断に迷う場合は blue（デフォルト）を選択。スタッフが手動で変更可能。

#### 生成手順

1. `config-schema.json` のスキーマに準拠
2. `outputMode` に応じたサンプルをベースにする:
   - proposal_single → `samples/config-v2-megasolar.json` 等を参考
   - proposal_multi → `samples/config-v2-backup-multi.json` を参考
3. `manualPricing` が不要（A〜D クラス）の場合:
   - `src/v2/pricing-calculator.js` の関数を使って料金を計算
   - 計算結果を config に反映
4. `manualPricing` が必要（OVER クラス）の場合:
   - Step 4 のテンプレ値ガイドラインに従い初期値を設定
   - `⚠️ 個別見積り案件です。料金はスタッフが確認・調整してください。` と報告
5. 全フィールドを埋めたら JSON として妥当性を確認（パースエラーがないこと）

#### proposal_multi 固有の構造

4プラン（A1, A2, B1, B2）を `pricing.plans` 配列に格納する:

| プラン | 記帳代行 | 自計化 | 相談サポート |
|--------|---------|--------|------------|
| A1 | ✓ | - | - |
| A2 | ✓ | - | ✓ |
| B1 | - | ✓ | - |
| B2 | - | ✓ | ✓ |

年次料金は `pricing.commonAnnual` に全プラン共通で格納。
自計化プランの初年度費用は `pricing.selfBookkeepingFirstYearOnly` に格納。

#### config 生成後の報告フォーマット

```
## config.json 生成結果

| 項目 | 値 |
|------|-----|
| お客様名 | 〇〇株式会社 |
| outputMode | proposal_single / proposal_multi |
| salesClass | A / B / C / D / OVER |
| manualPricing | true / false |
| 月次料金（税別） | XX,XXX円 |
| 年次料金（税別） | XXX,XXX円 |
| 課題数 | X |
| サービス数 | X |
| 出力先 | references/pricing/config_xxx.json |

### 確認事項
- （不明だった項目や要確認事項があればここに記載）
```

### Step 8: 提案書生成（フェーズ3で実装予定）

config.json をもとに提案書（.docx）を生成する。

```bash
node src/v2/create-proposal.js <config.json> [output.docx]
```

※ フェーズ3で `src/v2/create-proposal.js` を実装後に有効化。
現時点ではStep 7（config.json生成）までが本スキルの範囲。

## 重要な原則

設計書（`docs/proposal-design-v1.md` セクション11）に基づく:

1. **データと表現の分離**: 料金計算（pricing-calculator）、案件情報（config）、出力（テンプレート）は完全分離
2. **手動入力の優先**: スタッフが最終調整できる箇所を残す（出精値引き・紹介値引き・manualPricing）
3. **失敗の見える化**: 1億円超やデータ不足は警告し、勝手に推測しない
4. **既存運用への配慮**: `G:\共有ドライブ\06_見込み客\{YYYYMMDD}_{お客様名}\` の運用フローを尊重
5. **議事録は唯一の信頼源ではない**: 抽出できなかった項目は「不明」と明示し、スタッフに判断を委ねる

## 呼び出し例

以下のような指示で本スキルが発動する:

```
# 基本的な呼び出し
「提案書を作って」
「見積書を作成して」
「〇〇株式会社の提案書を作成」

# パス指定あり
「G:\共有ドライブ\06_見込み客\20260401_BackUP\ の議事録から提案書を作って」

# 議事録ファイル直接指定
「この議事録から config.json を作って」（ファイルパスが前の会話で共有されている場合）

# 既存 config の修正
「config_kanuu.json の値引き額を調整して」
```

## テンプレート構造（出力される提案書の構成）

提案書は以下のセクションで構成される（全モード共通）:

1. **表紙** — 会社名 / 提案日 / 事務所名 / コンセプト
2. **はじめに** — 敬意ある挨拶 + 顧客の事業概要（`proposal.introduction`）
3. **貴社の課題認識** — 表形式（課題 / 詳細）（`challenges`）
4. **サービス内容** — 表形式（サービス項目 / 説明）（`services`）
5. **料金プラン** — モードにより構造が異なる（下記参照）
6. **備考・補足事項** — 経理サポートについて / 相談サポートについて / コミュニケーション体制 / 有効期限・お支払条件
7. **ご契約後の流れ** — STEPテーブル（`contractFlow`）
8. **当事務所の概要** — 事務所情報（テンプレート固定）

### 料金プランの構造

#### proposal_single（お任せ型）
月次料金テーブル + 年次料金テーブルの2つ。
それぞれ項目・内訳・金額を表形式で表示。値引きは独立行として表示。

#### proposal_multi（4択型）
プランA1・A2の月次料金 → A1/A2共通の年次料金 → プランB1・B2の月次料金 → B1/B2共通の年次料金 → 初年度のみの年次料金（自計化プラン） → 年間費用サマリー表。

詳細なテーブルレイアウトは `docs/proposal-design-v1.md` セクション6.2 を参照。
