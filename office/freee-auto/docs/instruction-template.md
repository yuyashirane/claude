# Claude Code 修正系タスク 指示書テンプレート

## 使い方

修正系タスクをClaude Codeに依頼する際は、このテンプレートに沿って指示書を作成してください。
テンプレートに沿うことで、過去の失敗パターン（独自判断による意図しない追加、実物確認漏れ、影響範囲の見落とし等）を回避できます。

---

# [タスク名]

## 1. 背景

なぜこの修正が必要か、簡潔に1-3段落で記述。

例:
> `monthly-report-generator.js` が出力する Excel レポートの「PL月次推移」シートに、
> 「法人税・住民税及び事業税」行が `#VALUE!` 表示になるバグが発見された。
> 原因は Excel HYPERLINK 関数の URL 文字数制限（255文字）超過。
> 決算レビューで使用するため、優先度高で修正が必要。

## 2. 現象

具体的な症状を記述。再現手順、エラーメッセージ、影響範囲を明示。

例:
> 474381（あしたの会計事務所）の 2026-03 レポートで以下を確認:
> - 「法人税・住民税及び事業税」行の全月セルが `#VALUE!` 表示
> - 他の科目（役員報酬、給料手当など）は正常表示

## 3. 原因（推定 or 確定）

原因が特定できている場合は記述。不明な場合は「Step 1で調査」と書く。

例:
> Excel HYPERLINK 関数の URL 引数 255 文字制限を超過。
> 「法人税・住民税及び事業税」のような長い科目名で URL が 275 文字となり、Excel が `#VALUE!` を返す。

## 4. 修正方針（確定済み or 候補）

修正方針を記述。確定済みの場合はその方針、複数候補がある場合はそれぞれのメリット・デメリット。

**重要**: 使用すべきパラメータ・関数を**明示的に列挙**する。これにより Claude Code が「良かれと思って」追加することを防ぐ。

例:
> ### 確定方針: 総勘定元帳URLに変更
>
> ユーザー検証済みの正しいURL形式:
> ```
> https://secure.freee.co.jp/reports/general_ledgers/show?name=...&start_date=...&end_date=...&fiscal_year_id=...
> ```
>
> **必須パラメータ（4つのみ）**:
> - `name` (URLエンコード済科目名)
> - `start_date` (YYYY-MM-DD)
> - `end_date` (YYYY-MM-DD)
> - `fiscal_year_id`
>
> **禁止パラメータ（絶対に追加しない）**:
> - `adjustment` — 期間指定が無効化される
> - `page`, `per_page`, `order_by`, `direction` — freeeデフォルトで代替可能
> - `account_item_id` — `name` と機能重複
> - `source_type`, `gl_summation_method`, `straddled_fiscal_year` — freeeが自動判定

## 5. 修正手順

### Step 1: 現状調査

```bash
# 該当箇所を grep で特定
grep -n "..." path/to/file.js

# 該当ファイルを読む
cat path/to/file.js
```

確認すべきポイント:
1. ...
2. ...
3. ...

### Step 2: 影響範囲の確認

```bash
# 関連する呼び出し元を検索
grep -rn "functionName" src/
```

確認すべきポイント:
- どのシートに影響するか
- 既存テストへの影響
- 後方互換性

### Step 3: 調査結果の報告（コード変更前）

以下のフォーマットで報告してください:

```
■ 該当ファイル・関数:
  ファイル:
  関数名:
  該当行: L XX-XX

■ 影響範囲:
  - 影響するシート/関数:
  - 既存テストへの影響:

■ 提案する修正方法:
  方法A: ...
  方法B: ...
  推奨: 方法X（理由: ）
```

**この時点で必ずユーザー確認を取ること。承認を得てから Step 4 に進む。**

### Step 4: 実装

ユーザー承認後、以下を実施:

1. **対象ファイルの修正**
   - 修正方針通りに実装
   - 指示書に明記されたパラメータのみ使用
   - **「良かれと思って」追加しない**

2. **既存テストの期待値更新**（必要なら）
   - テスト期待値を変える場合は仕様変更とみなし、ユーザーに確認

3. **新規テストの追加**（再発防止）
   - 単体テストで境界値をチェック
   - 例: URL長チェック、null/undefined チェック、エッジケース

### Step 5: 検証

```bash
# 全テスト実行
npm test
node tests/test-xxx.js  # 個別テスト

# 実データで確認
node src/verify/monthly-checker.js --company XXX --month YYYY-MM --no-dry-run

# 検証スクリプトで詳細確認（必要に応じて）
node -e "
// 生成されたファイルを開いて、特定の項目を確認
"
```

検証項目:
- [ ] 全テスト通過
- [ ] 実データで生成成功
- [ ] エッジケース（最長文字列等）の確認
- [ ] 既存機能への影響なし

### Step 6: 報告

以下のフォーマットで報告してください:

```
■ 原因:
  （Step 3 で特定した原因）

■ 修正内容:
  ファイル:
  修正行数:
  修正箇所:

■ テスト結果:
  npm test: XXX件通過
  個別テスト: XX件通過
  新規追加テスト: X件通過

■ 実データ検証:
  対象会社:
  対象月:
  確認項目1: ✅/❌
  確認項目2: ✅/❌

■ 残る懸念点:
  （あれば記載、なければ「なし」）

■ 生成ファイル:
  reports/.../...
```

## 6. ユーザー側で確認していただく動作

実装完了後、ユーザーに以下を確認してもらいます:

1. **具体的な確認手順**を番号付きで明示
2. **期待される結果**を具体的に記述
3. **失敗時の報告フォーマット**を指示

例:
> 修正完了後、以下を確認お願いします:
>
> 1. Excel ファイルを開く
> 2. PL月次推移シートを開く
> 3. 法人税・住民税及び事業税の3月セル（G42）をクリック
> 4. ブラウザで freee の総勘定元帳が開かれる
> 5. 表示される期間が「2026年3月1日〜3月31日」であること（1年度分ではないこと）
> 6. 表示される科目が「法人税・住民税及び事業税」であること
>
> すべて期待通りの場合は「OK」とお知らせください。
> 1つでも異なる場合は、どのステップで何が違ったかを教えてください。

## 7. 注意事項

- **後退なし原則**: 既存の正常動作している機能を壊さない
- **指示書のパラメータ厳守**: 良かれと思って追加しない
- **freee書き込みなし**: 読み取りのみ
- **段階的アプローチ**: Step 3 で必ずユーザー確認
- **実物確認**: テスト通過 ≠ 動作OK
- **再発防止**: 単体テストで境界値チェックを追加

## チェックリスト（指示書作成時）

指示書を作成したら、以下を確認してから Claude Code に渡してください:

- [ ] 1. 背景は明確か（なぜこの修正が必要か）
- [ ] 2. 現象は具体的か（再現手順、エラーメッセージ）
- [ ] 3. 原因の推定（または「Step 1で調査」と明記）
- [ ] 4. **使用すべきパラメータ・関数を列挙したか**
- [ ] 4. **禁止パラメータがあれば明記したか**
- [ ] 5. 影響範囲調査の手順を明示したか
- [ ] 5. **Step 3 で「ユーザー確認の停止ポイント」を設けたか**
- [ ] 6. 検証手順に「実物確認」を含めたか
- [ ] 6. 単体テスト追加（再発防止）を含めたか
- [ ] 7. 報告フォーマットを明示したか
- [ ] 8. ユーザー側の動作確認手順を番号付きで明示したか

すべて ✅ になったら、指示書として完成です。

## 過去の失敗事例（このテンプレートが防ぐもの）

### 事例1: adjustment=all バグ（2026-04-07）

**問題**: 仕訳帳→総勘定元帳リンク変更時、Claude Codeが独自判断で `adjustment=all` を追加した。
このパラメータは期間指定を無効化するため、リンクをクリックすると1年度分が表示される問題が発生。

**このテンプレートでの防止策**:
- セクション 4「修正方針」で **使用すべきパラメータと禁止パラメータを明示的に列挙**
- セクション 5 Step 4「実装」で **「指示書に明記されたパラメータのみ使用」「良かれと思って追加しない」を強調**

### 事例2: PL月次推移バグの見落とし（2026-04-07）

**問題**: 19個の集計行が `accounts["undefined"]` に上書きされていたバグ。
全自動テストは通過していたが、実物の Excel を開けば一目瞭然だった。

**このテンプレートでの防止策**:
- セクション 5 Step 5「検証」で **実データでの生成と実物確認を必須化**
- セクション 6「ユーザー確認」で **具体的な確認手順を明示**

### 事例3: name= 削除案の不備（2026-04-07）

**問題**: URL 短縮のため `name=` パラメータを削除する案を提案したが、これは勘定科目フィルターに必須だった。
机上の文字数計算だけで判断し、実務的な意味を確認していなかった。

**このテンプレートでの防止策**:
- セクション 5 Step 3「調査結果の報告」で **修正方針を提案し、ユーザー確認を取る**
- セクション 7「注意事項」で **「実物確認」を強調**

### 事例4: fiscal_year_id の必要性見落とし（2026-04-07）

**問題**: 総勘定元帳URLの最小構成を試した際、`fiscal_year_id` がなくても表示されるように見えたが、実際には期間指定が効かず1年度分が表示されていた。
ユーザーの「最初の動作確認」が不十分で、間違った前提で実装を進めた。

**このテンプレートでの防止策**:
- セクション 6「ユーザー確認」で **期待される結果を具体的に記述**
- セクション 5 Step 5「検証」で **エッジケースの確認を含める**
