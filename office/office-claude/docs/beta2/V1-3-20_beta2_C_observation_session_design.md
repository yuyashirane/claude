## §1. このセッションの目的（最重要文）

> **β2-C で実装した構造（Finding / FindingGroup / observations）が、複数社で「説明可能な世界」を提示しているかを観察する。**

### 核心の方針

- 「正しいか」ではなく **「世界の見え方が一貫しているか」** を見る
- 「綺麗な分布」ではなく **「説明できる分布」** を求める
- 検出ロジックの検証ではない（β2-B で完結済み）

### β2-A〜β2-B との性質の違い

| フェーズ | 性質 |
|---|---|
| β2-A | データ構造方針の確定 |
| β2-B | 検出ロジック（5 分類体系）の正しさ |
| β2-C 実装 | 出力構造（Finding / Group / observations）の整備 |
| **β2-C 観察セッション**（本書） | **出力構造の説明可能性の検証** |

β2-C 実装で「検出するAI」から「観察できるAI」へ転換した。本セッションはその観察能力を 3 社で検証する。

### スコープ外（明示）

- ❌ 474381 の観察（データ整備後に別セッション、観察純度の保護）
- ❌ コード変更（観察フェーズでは戦略 Claude / 悠皓さんのみ稼働、Claude Code は出番なし）
- ❌ 件数妥当性の議論（β2-D 範疇）
- ❌ manual_journals 取り込みの判断（β2-D 範疇）
- ❌ T 番号妥当性 / 少額特例 / 公共交通費特例（β2 全体スコープ外）
- ❌ Excel 統合 / 親子行レイアウト（β2-E / β3 範疇）

## §3. 観察項目（基本 3 項目 + 条件付き深掘り）

### 基本観察（全 3 社で実施）

#### 1. classification_counts
6 値の分布を確認：
- qualified_but_transitional_tax
- nonqualified_but_full_deduction_tax
- partner_unknown
- expected_transitional_tax
- expected_full_deduction_tax
- none

#### 2. groups
- findings_count（3 分類別）
- **主要分類のサンプル 1 件**（warning Finding のうち件数が多い分類）
- 必要に応じて他分類のサンプルも確認

主要分類とは、その社で **最も件数が多い Finding 化対象分類**。3525430 では nonqualified_but_full_deduction_tax（2 件）。

#### 3. observations.partner_unknown_breakdown
- absorbed_into_nonqualified
- remaining_partner_unknown
- 解釈 X 推定吸収の効きが社ごとに異なるかを観察

### 深掘り（違和感が出た会社のみ実施）

違和感が出るとは：
- 戦略 Claude の読み取り型 5 項目で「説明できない」が出た
- 悠皓さんが税理士視点で「この値はおかしい」と感じた
- 不変条件が崩れた

#### 深掘り項目

##### a. raw 8 フィールド
実データの finding 例で 8 フィールドが実用的か：
- tax_label / tax_code / debit_amount / partner / description
- transaction_date / source / is_qualified_invoice
- 「この raw だけで実務スタッフが確認に動けるか」が確認軸

##### b. message テンプレートの可読性
実務スタッフ目線で：
- 修正アクションが明確か
- partner 空時「取引先不明」表示が混乱を招いていないか
- テンプレートの分類別差分が伝わるか

##### c. 不変条件 2 つの再確認
3 社それぞれで成立するか機械的にチェック：
- ① `sum(classification_counts.values()) == total_rows`
- ② `sum(group.findings_count for group in groups) == findings_count`

3525430 では実装時に確認済み（2064 == 2064 / 2 == 2）。12243357 / 10794380 で同様に成立するかは観察セッションで初確認。

## §4. 判断基準（観察前に確定する重要項目）

### パターン A（進んでよい）

#### 条件
- **不変条件**：3 社全社成立
- **classification**：各社で意味のある分布（少なくとも 2 分類以上が非 0）
- **observations**：説明可能（違和感が言語化できる）

#### 核心
**「綺麗」ではなく「説明できる」**

partner_unknown が 0 件でも「なぜ 0 件か説明できる」なら A、説明できないなら B。
qualified_but_transitional_tax が突出して多くても「なぜ多いか説明できる」なら A。
**数値の分布ではなく、説明可能性が判断基準**。

#### アクション
- β2-D 着手判断書を作成
- β2-D 設計フェーズへ進む（manual_journals / 3 社観察項目 / 474381 fetch / observations 追加項目）

### パターン B（調整してから進む）

#### 条件
- **不変条件**：成立
- **classification**：偏りあり
- **observations**：説明が曖昧 or 過不足あり

#### 核心
**「壊れてはいないが、解像度が足りない」**

不変条件は守られている（構造は健全）。だが、observations の partner_unknown_breakdown だけでは説明しきれない違和感がある。たとえば：
- 「partner_unknown の内訳が分かっても、qualified_but_transitional_tax が突出する理由が見えない」
- 「nonqualified_but_full_deduction_tax の中で description が空のものが多い理由が分からない」

#### アクション
- 調整論点を整理
- β2-C 微修正の設計フェーズに戻る（observations 追加 / message 微調整 / 観察項目追加など）
- 設計が確定したら Claude Code に GO

### パターン C（止める）

#### 条件
- **不変条件**：崩れる
- 同分類内で意味が揺れる（message 破綻含む）
- groups を見ても「何を意味しているか分からない」

#### 核心
**「構造が嘘をついている状態」**

不変条件①（classification_counts の合計 == total_rows）が崩れていれば classify_transaction の挙動異常。不変条件②（group.findings_count の合計 == findings_count）が崩れていれば find_groups または FINDING_TARGET_CLASSIFICATIONS の不整合。message が破綻していればテンプレート差し込みの設計違反。

#### アクション
- β2-C 微修正の指示書を即作成
- Claude Code に即 GO（深刻度高）
- 構造再検証 → 再観察

## §6. 戦略 Claude の読み取りの型（§4 判定とは別物）

### 読み取りの型（5 項目）

戦略 Claude は、各社の基本 3 項目を確認した後、**以下の形式に限定して読み取り**を行う：

1. **この会社では、どの分類が多い／少ないか**
2. **その分布は、会社の既知の性質から説明できるか**
3. **observations は、その説明を補助しているか**
4. **groups のサンプルは、分類名と意味が一致しているか**
5. **追加で深掘りすべき点があるか**

### 重要な制約

**戦略 Claude はこの時点で A/B/C の最終判定をしない。**

理由：
- 戦略 Claude が A/B/C を提示すると、悠皓さんの判断が引っ張られる
- 戦略 Claude は構造的な読み取りに徹する役割
- A/B/C の確定は税理士視点が必要（悠皓さんの責務）

ただし、§5 の「3 社横断の読み取り」では **「パターン A / B / C のどれに近いか」の仮説提示**は許容する（最終判定ではなく、論点整理のため）。

### 読み取りの例（型のイメージ）

```
12243357 の読み取り（仮想例）：

1. どの分類が多い／少ないか
   → expected_full_deduction_tax と none が多い、qualified_but_transitional_tax が 0 件

2. その分布は会社の既知の性質から説明できるか
   → β2-B で「全件 qii=False」と観察済みなので、qualified_* 分類が 0 件なのは説明可能

3. observations は説明を補助しているか
   → partner_unknown_breakdown の absorbed が多い場合、qii=False の事業者比率が高いことを補助している

4. groups のサンプルは分類名と意味が一致しているか
   → nonqualified_but_full_deduction_tax の message が「非適格事業者ですが…」で始まる、一致

5. 追加で深掘りすべき点
   → expected_full_deduction_tax の比率が他社と大きく異なる場合、深掘り候補
```

### 型を守る理由

- 縛りすぎない（深掘りや仮説の余地を残す）
- 判断誘導を防ぐ（A/B/C を勝手に決めない）
- 3 社横断の比較可能性を担保（読み取り項目が揃う）

## §7. 観察セッションの進め方

### 7.0 観察の心構え（最重要）

> **観察は「結論を出すため」ではなく、「説明可能性を確認するため」に行う。**

この一文を観察セッション中に何度でも参照する。特にパターン B（壊れてないが解像度不足）が出たときに有効：

- 「結論を出さなければ」と焦ると、無理に A や C に振り分けたくなる
- 観察の本来の目的は「説明できるか」を確認すること、結論を出すことではない
- B のまま留めることも、観察の正しい結果

戦略 Claude も悠皓さんも、この心構えを共有して観察に臨む。

### 7.1 事前準備（次セッション開始前）

#### 戦略 Claude の準備

3 社それぞれを Claude Code で run.py 実行し、JSON 出力を取得する。

```bash
# 3 社分を先取得（観察セッション開始前）
PYTHONIOENCODING=utf-8 py -3 skills/verify/V1-3-rule/check-invoice-registration-status/run.py --company-id 3525430 --target-month 2025-12 > observation_3525430.json
PYTHONIOENCODING=utf-8 py -3 skills/verify/V1-3-rule/check-invoice-registration-status/run.py --company-id 12243357 --target-month 2025-12 > observation_12243357.json
PYTHONIOENCODING=utf-8 py -3 skills/verify/V1-3-rule/check-invoice-registration-status/run.py --company-id 10794380 --target-month 2025-12 > observation_10794380.json
```

これは観察前の「データ収集」段階で、**コード変更ゼロ**。Claude Code はこの実行のみで稼働終了。

#### 悠皓さんの準備

3 社の既知情報を再確認：
- 3525430：β2-B 観察結果（findings_count=2、解釈 X が 2 件適用）
- 12243357：β2-B 観察結果（全件 qii=False）
- 10794380：β2-B 観察結果（qii=True × IRN なし 13 件）

### 7.2 観察セッション本体

```
1. 3525430 の基本 3 項目を確認
   → 戦略 Claude が読み取り型 5 項目で出力
   → 悠皓さんがコメント
   → 違和感があれば深掘り、なければ次へ

2. 12243357 の同手順
3. 10794380 の同手順

4. 3 社横断の読み取り（戦略 Claude）
   → 共通傾向 / 差異 / 説明可能性の総合評価
   → A / B / C のうちどれに近いかの仮説提示（最終判定はしない）

5. 悠皓さんが A/B/C を確定
   → 判断理由を観察ログに記録
   → 次アクションを決定
```

### 7.3 想定所要時間

- **基本観察のみ**：60 分（1 社あたり 15-20 分 × 3 社 + 横断 10 分）
- **深掘りが発生**：+30 〜 60 分（深掘り対象社数 × 深掘り項目数による）

90 分を超える場合は休憩を挟む。集中力が落ちると判断の質が下がる。

### 7.4 αモード（事前一括取得）の運用

質問 1 で確定した方針：

```
3 社分を先に取得
  → 観察は 1 社ずつ
  → 深掘りは必要な会社だけ
```

これにより：
- 1 社目の違和感で観察軸を変えすぎず、比較可能性を保つ
- データ取得の中断によるセッション分断を防ぐ
- 深掘りが必要な社だけ追加調査することで時間を節約
