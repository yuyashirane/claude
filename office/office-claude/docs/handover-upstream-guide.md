## 1. プロジェクトの目的

### 最終的に実現したいこと

税理士事務所（あしたの会計事務所税理士法人）の経理業務を、Claude Code + freee会計 + Kintone の3システム連携で自動化・効率化すること。具体的には以下の4業務：

1. **明細→取引登録**: freeeの未処理明細や顧問先から届くExcel/CSVを読み込み、仕訳候補を自動生成してfreeeに登録する
2. **帳簿チェック**: freeeの記帳内容を定期的にチェックし、消費税区分の誤り・異常値・固定資産化漏れ等を検出する
3. **証憑整理**: 領収書・請求書等の画像/PDFをOCRで読み取り、分類・リネーム・freeeファイルボックスへ整理する
4. **証憑→取引登録**: 証憑からの仕訳候補生成（上記①③の融合）

### なぜやるのか

- **人手不足**: 経理担当（特に若手）の作業負荷が大きい
- **判断のバラつき**: 科目判定や消費税区分の判断が人によって異なる
- **チェック漏れ**: 月次・決算時のチェックが属人的で、見落としが発生する
- **教育コスト**: 若手担当者への教育に時間がかかる

### 解決したい業務課題

| 課題 | 現状 | 目標 |
|------|------|------|
| 未処理明細の処理 | 全件手動で科目・税区分を判断 | 高確度は自動登録、中低確度のみ人確認 |
| 消費税区分の確認 | 目視チェックで見落としあり | 12ルール（R01〜R12）で網羅的に自動チェック |
| 帳簿の定期チェック | 属人的、決算直前にまとめて | 月次で自動チェック、指摘事項をKintone管理 |
| 証憑の整理 | 手作業でファイル名変更・分類 | OCR→自動分類→freeeファイルボックスへ |

## 2. プロジェクトの基本方針

### 6つの設計原則（合意済み・確定事項）

| # | 原則 | 具体的な意味 |
|---|------|-------------|
| 1 | **freeeファースト** | freeeの標準機能（自動登録ルール、口座同期）で処理できるものはfreeeに任せる。Claude Codeは「freeeで処理されなかった残り」だけを扱う |
| 2 | **人確認は必要最小限** | 高確度（75点以上）は自動処理、中低確度と例外のみ人が確認。ただし初期は全件確認を経てから段階的に自動化の範囲を広げる |
| 3 | **Kintoneが確認フローの中心** | 要確認案件はKintoneに集約。担当者確認→承認→freee反映→辞書改善の流れをKintone上で管理する |
| 4 | **説明可能な判定** | AIが出した判定には必ず理由を付与する（どのキーワードにマッチしたか、スコア内訳はどうか、どの税ルールが適用されたか） |
| 5 | **安全性重視** | freeeへの書き込みはドライランが標準。本番登録は明示的に許可した場合のみ。ログは全件記録 |
| 6 | **学習可能な構造** | 人の修正結果をキーワード辞書・取引先辞書の改善に反映し、精度を継続的に向上する |

### Claude Codeに任せる範囲（確定）

- freee APIからのデータ取得（読み取り）
- 明細の標準化（日付変換、金額正規化、全角→半角）
- 勘定科目・消費税区分の推定（辞書・ルールベース）
- 信頼度スコアの算出
- 振り分け判定（自動登録 / Kintone確認 / 除外）
- Excelレポートの出力
- 帳簿チェック（消費税R01〜R12、固定資産、異常値検出）
- Kintoneへのレコード登録（確認案件の送付）

### 人が判断すべき範囲（確定）

- **消費税区分の最終判断**: AIは候補を出すが、最終確定は税理士が行う
- **固定資産か修繕費かの判断**: 10万円以上の取引は必ず人が確認
- **初回取引先の科目判定**: 過去実績がない取引先は人が確認
- **高額取引の確認**: 一定額以上（現在10万円）は人が確認
- **辞書・ルールの更新承認**: AIが提案するが、反映は人が承認する
- **freeeへの本番書き込みの許可**: ドライラン→確認→本番の流れを人が管理

### 自動化しすぎてはいけない部分

1. **freee本番への自動書き込み**: 初期段階では全件Kintone確認を経由すべき。十分な実績（High正答率95%以上）が溜まってから段階的に開放
2. **辞書の自動更新**: AI提案→人承認→反映の3ステップを崩さない
3. **消費税の最終判定**: R01〜R12は「指摘」であり「確定」ではない
4. **顧問先のデータを使ったテスト**: テストは自社アカウントのみ。顧問先データでのテストは本番運用開始後

### 安全性・正確性のための前提

- freeeへの書き込みは必ず `--dry-run` でテストしてから実行
- 1バッチ最大50件の制限（レートリミット対策 + 事故防止）
- 処理結果は全件JSON + Excelでログ保存
- テスト用事業所: **あしたの会計事務所税理士法人**（freee ID: 474381、9月決算）
- 実運用テスト用: **株式会社デイリーユニフォーム**（freee ID: 10794380、5月決算）

## 3. 想定している全体業務フロー

### 3-1. 7段階パイプライン（確定）

```
INPUT → NORMALIZE → CLASSIFY → REVIEW → REGISTER → VERIFY → LEARN
書類取込  明細標準化   仕訳判定   人間確認   freee登録  帳簿チェック 学習改善
```

### 3-2. 入力→出力の全体フロー

```
【入力ソース】                    【Claude Code処理】              【出力先】
─────────────                    ──────────────                   ────────
freee未処理明細 ─────┐
                     ├─▶ NORMALIZE ─▶ CLASSIFY ─▶ ROUTING
顧問先Excel/CSV ────┘     (標準化)     (判定)      (振り分け)
                                                      │
                                          ┌───────────┼───────────┐
                                          ▼           ▼           ▼
                                     自動登録      Kintone確認    除外
                                    (75点以上)    (45-74:若手)   (ログ記録)
                                    freee API    (0-44:経験者)
                                                      │
                                                      ▼
                                                 人間レビュー
                                                 承認/修正/却下
                                                      │
                                          ┌───────────┤
                                          ▼           ▼
                                     freee登録     辞書改善提案
                                                  (LEARNモジュール)

                                    ※ 全処理結果 → Excelレポート出力
```

### 3-3. 帳簿チェックフロー（別ライン・Phase 2）

```
freee BS/PL/取引データ
    ↓
帳簿チェック（32手続、消費税R01〜R12、固定資産、人件費等）
    ↓
┌─────────────┬──────────────┐
▼             ▼              ▼
Excelレポート   Kintone指摘     ログ
(全件)        (🔴🟡のみ)     (処理記録)
```

### 3-4. Kintoneの位置づけ

Kintoneは **人間確認のハブ** として機能する。3つのアプリを想定：

| アプリ | 用途 | 状態 |
|--------|------|------|
| 仕訳レビュー | 仕訳候補の確認・承認・修正 | 仕様書あり、アプリ未作成 |
| 帳簿チェック | チェック指摘事項の管理 | 仕様書あり、アプリ未作成 |
| 学習フィードバック | 辞書・ルール改善提案の管理 | 仕様書あり、アプリ未作成 |

### 3-5. 例外処理の考え方

| 例外パターン | 処理 |
|-------------|------|
| freeeのルールでマッチ済み（rule_matched=true） | Claude Codeは処理しない（freeeファースト） |
| 除外キーワード該当（振替、相殺、取消等） | 除外ログに記録、仕訳登録しない |
| 高確度だが消費税指摘フラグあり | 自動登録せずKintone送付 |
| 高確度だが10万円以上 | 自動登録せずKintone送付（固定資産確認） |
| 高確度だが勘定科目が「雑費」 | 自動登録せずKintone送付（科目不明の可能性） |
| 日付パースエラー | 無効明細としてエラーログに記録 |
| freee API エラー | リトライ後もダメならエラーログ＋アラート |

## 4. 現時点で決まっている要件

### 4-1. 確定済み事項

#### パイプラインアーキテクチャ
- 7段階パイプライン: INPUT → NORMALIZE → CLASSIFY → REVIEW → REGISTER → VERIFY → LEARN
- 各段階は独立したモジュールとしてsrc/配下に実装

#### 信頼度スコア
- 5要素×重み付け → 100点満点
  - キーワード辞書マッチ: 30pt
  - 過去仕訳パターン一致: 30pt（現在プレースホルダー、0ptで固定）
  - 金額の妥当性: 15pt
  - 消費税ルール明確さ: 15pt
  - 摘要の情報量: 10pt
- 閾値: High(75+) / Medium(45-74) / Low(0-44) / 除外

#### 消費税区分ルール
- R01〜R12の12ルールで判定（`references/tax/tax-classification-rules.md`）
- ルールはフラグ（指摘）であり、最終判断は人が行う

#### インボイス制度対応
- 5区分: 適格 / 非適格(経過80%) / 非適格(経過50%) / 不要 / 要確認
- 経過措置の期限管理: 80%=2026/9/30まで、50%=2029/9/30まで

#### 除外ルール
- 除外キーワード: 振替, 振込, 相殺, 戻入, 取消, キャンセル, 口座間, 立替, 預り, 仮受, 仮払
- freee内部振替、重複取引も除外

#### 勘定科目辞書
- 22科目のキーワード辞書が実装済み（`src/classify/account-matcher.js` 内にハードコード）
- 将来的には `references/dictionaries/account-keywords.md` に外部化予定

#### freee勘定科目IDマッピング
- 28科目のfreee API IDが確認済み（`src/classify/unprocessed-processor.js` 内の `FREEE_ACCOUNT_IDS`）
- ⚠️ このIDは事業所ごとに異なる。現在はあしたの会計事務所のIDがハードコードされている

#### 消費税コードマッピング
- 9区分のfreee税コードが確認済み（`TAX_CLASS_TO_CODE`）

### 4-2. ほぼ確定だが微調整の余地がある事項

| 項目 | 現状 | 調整の余地 |
|------|------|-----------|
| 自動登録の閾値 75点 | テストデータで検証済み | 実運用データで閾値を再評価する必要あり |
| 1バッチ最大50件 | 安全策として設定 | 運用実績を見て増減可能 |
| 高額取引の閾値 10万円 | 固定資産の少額減価償却を意識 | 30万円（中小企業の特例）に変更するか検討 |
| API呼び出し間隔 500ms | レートリミット対策 | freee APIの負荷状況次第で調整 |
| 「過去仕訳パターン一致」スコア 0pt | 未実装のためプレースホルダー | 実装後は30pt満点で評価 |

### 4-3. 技術・運用の前提ルール

- **言語**: Node.js (CommonJS)
- **freee API接続**: freee MCP（Claude Code経由）
- **Kintone接続**: @kintone/rest-api-client (npm)
- **Excel出力**: ExcelJS (npm)
- **OCR**: Claude Vision API（Phase 3以降）
- **テスト**: 手動テストスクリプト（`tests/` 配下）、フレームワーク未導入
- **実行モデル**: オンデマンド実行（常駐プロセスなし）
- **状態管理**: ファイルベース（data/, logs/, tmp/）

## 7. ルール・設計思想

### 7-1. 命名規則（確定・暗黙合意）

| 対象 | ルール | 例 |
|------|--------|-----|
| ファイル名 | kebab-case | `account-matcher.js`, `format-standardizer.js` |
| 関数名 | camelCase | `classifyTransaction()`, `standardizeRows()` |
| 定数 | UPPER_SNAKE_CASE | `FREEE_ACCOUNT_IDS`, `TAX_CLASS_TO_CODE` |
| フォルダ名 | kebab-case | `freee-auto`, `source-document-intake` |
| Kintoneフィールドコード | snake_case | `confidence_score`, `review_status` |

### 7-2. 実装方針

- **CommonJS** (`require/module.exports`) を使用（`"type": "commonjs"` in package.json）
- 各モジュールは **純粋関数** を目指す（入力→出力が明確、副作用を最小化）
- 外部API呼び出し部分は **モック差し替え可能** に設計（テスト容易性）
- ハードコードされた辞書・ルールは将来的に外部ファイルに切り出す

### 7-3. 例外処理方針

- 日付パースエラー → 無効明細としてエラーリストに格納、処理続行
- 金額パースエラー → 0円として扱い、スコアを低くする
- freee APIエラー → エラーログ記録、リトライ後もダメならスキップ
- Kintone APIエラー → エラーログ記録、ドライランJSONに出力

### 7-4. ログ出力方針

- 処理結果JSON: `tmp/processing_result_{timestamp}.json`
- 登録ログJSON: `tmp/registration_log_{timestamp}.json`
- Excelレポート: `reports/processing_report_{company_id}_{timestamp}.xlsx`
- Kintone送信ログ: `data/{company_id}/{date}/kintone_send_log.json`

### 7-5. セキュリティ方針

- `.env` ファイルはGit管理外（`.gitignore`対象）
- APIトークンは環境変数経由で読み込み
- `credentials.json`, `token.json` はGit管理外
- freee MCPの認証はClaude Codeのセッション内で管理

### 7-6. 顧客情報・本番データの取り扱い方針

- **テスト**: 自社アカウント（あしたの会計事務所、ID: 474381）のデータのみ使用
- **顧問先データ**: 本番運用開始後に顧問先の明示的な同意を得てから使用
- `data/` 配下の分析結果はGit管理対象だが、機密データ（生の取引明細等）は `tmp/` に保存してGit管理外
- Kintoneのデータはクラウド上で管理（ローカルに永続保存しない）

### 7-7. freee本番更新時の確認ルール（確定）

1. **必ずドライラン（`--dry-run`）で事前確認** — 登録予定のペイロードを全件確認する
2. **Excelレポートで人間が内容を確認** — processing-report.js で出力したレポートを目視チェック
3. **少量テスト** — まず1〜3件を手動でfreeeに登録し、正しく反映されたか確認
4. **本番実行は `--no-dry-run` の明示が必要** — デフォルトはドライラン
5. **登録結果はログに全件記録** — registration_log_{timestamp}.json
6. **異常発生時は即座に停止** — 1件でもエラーがあれば残りをスキップする設計は未実装（要判断: U16）

## 11. ①担当者向けの実務上の注意点

### 11-1. 誤解しやすいポイント

| 誤解 | 実態 |
|------|------|
| 「AIが仕訳を確定してくれる」 | AIは **候補を提示** するだけ。最終確定は人が行う。特に消費税区分は税理士の判断が必要 |
| 「freeeに自動で登録される」 | デフォルトはドライラン。本番登録には明示的な許可が必要。初期は全件Kintone確認を推奨 |
| 「過去仕訳パターンでスコアが高くなる」 | 現在は未実装（0pt固定）。30pt分が0ptなので、Highランク（75pt以上）になる取引は非常に少ない |
| 「全顧問先に一括展開できる」 | freee勘定科目IDが事業所ごとに異なるため、1社ずつ設定が必要 |
| 「Kintoneアプリはコードで自動作成される」 | アプリの作成自体はKintone GUI or APIスクリプトで行う必要がある |

### 11-2. Claude Codeに丸投げすると危険な点

| 危険な丸投げ | なぜ危険か | 正しい対応 |
|-------------|-----------|-----------|
| 「freeeに登録して」（確認なし） | 誤った仕訳がfreeeに登録される | 必ずドライラン→Excelレポート確認→本番の手順を踏む |
| 「辞書を更新して」（影響範囲未確認） | 既存の判定結果が変わる可能性 | 更新前後でテスト結果を比較してから反映 |
| 「全顧問先のデータを処理して」 | 事業所ごとの設定差異を無視する | 1社ずつテストし、問題がないことを確認してから次の顧問先へ |
| 「自動で定期実行して」 | エラー発生時に気づかない | まずは手動実行で十分な実績を積む |

### 11-3. 要件が曖昧なまま進めると危険な点

| 曖昧な要件 | 起こりうる問題 | 先に決めるべきこと |
|-----------|---------------|-------------------|
| 「Excel取り込みに対応して」 | 顧問先ごとにフォーマットが異なり、汎用パーサーでは対応できない | まず1社のフォーマットを具体的に提示し、その1社に対応するパーサーを作る |
| 「精度を上げて」 | どの精度か不明（科目判定？税区分判定？全体スコア？） | 何を何%から何%に上げたいかを定義する |
| 「Kintoneと連携して」 | アプリが未作成、フィールド設計が未確定 | まずKintoneアプリを作成し、フィールドコードを確定してからコード側を実装 |

### 11-4. 人が責任を持って最終判断すべきポイント

1. **消費税区分の最終確定** — AIの指摘(R01〜R12)はあくまでフラグ。税理士が判断する
2. **固定資産 or 修繕費の判定** — 金額だけでは判断できない。内容の実態を確認する
3. **freee本番登録の許可** — ドライラン結果を確認し、問題なしと判断してから実行を許可する
4. **辞書・ルール変更の承認** — AIの改善提案を確認し、反映してよいか判断する
5. **新規顧問先への展開判断** — 十分なテスト実績があるか確認してから展開する
6. **自動登録の開放判断** — High正答率95%以上の実績が十分にあるか確認する

## 付録D: 現在の構想で不足している視点

| 不足している視点 | 影響 | 推奨アクション |
|-----------------|------|---------------|
| **複合仕訳（1取引に複数明細行）** | 現在は1取引=1科目=1行の前提。諸口・按分が必要な取引に未対応 | 実運用データで複合仕訳の出現頻度を確認し、対応優先度を判断 |
| **部門・セグメント対応** | freeeの部門機能を使っている顧問先に未対応 | 必要であればdealペイロードにsection_id（部門ID）を追加 |
| **仕訳の修正・削除フロー** | 誤って登録した仕訳の取り消し手順が未定義 | freee APIのPUT/DELETE対応 + Kintone上での取消フロー |
| **権限管理** | 誰がfreee登録を許可できるか、誰がKintoneで承認できるかが未定義 | Kintoneのプロセス管理で権限設定 |
| **バックアップ・リカバリ** | 辞書ファイルの誤更新時のリカバリ手順が未定義 | Git管理で対応（コミット単位でrevert可能） |
| **監査証跡** | 「誰が・いつ・何を・なぜ変更したか」の記録が不十分 | Kintoneのレコード履歴 + Git コミットログで対応 |
| **消費税の期間按分** | 年度をまたぐ取引（保険料の前払等）の消費税処理 | 発生頻度を確認してから対応方針を決定 |
| **通貨対応** | 外貨建て取引は現在想定外 | 外貨取引がある顧問先の有無を確認 |
