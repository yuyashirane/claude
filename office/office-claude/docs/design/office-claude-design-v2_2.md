# 1. プロジェクト概要
## 1.1 ビジョン
約240社のクライアントの記帳・月次チェック・決算業務をAI化し、窓口担当が高付加価値業務（税務相談・経営助言・顧客対応）に集中できる体制を構築する。
## 1.2 全体像
freee-autoプロジェクト（2026年3月〜）で蓄積した実績と知見をベースに、設計を一新して再構築する。

| 項目 | freee-auto（前身） | office-claude（新規） |
| --- | --- | --- |
| スコープ | freee会計のみ | 会計（freee）+ 税務（TKC）を統合 |
| 設計思想 | 行き当たりばったりの積み上げ | Agent Skills公式ベストプラクティス準拠 |
| Skill粒度 | 大粒度（66項目が1 Skillに集約） | 1 Skill = 1 判断 or 1 成果物 |
| 対象ユーザー | 所長のみ（Claude Code） | 全スタッフ（Cowork/デスクトップ版対応） |
| 自己改善 | 未実装 | Learnカテゴリで体系的に改善 |
| 実績 | テスト397件全通過、15モジュール66項目 | 設計段階（参照実装として活用） |

## 1.3 基本方針
freee-autoは凍結・運用継続。office-claudeをFrom Scratchで構築
freee-autoの判断ロジック・辞書・テストケースは「参照実装」として活用
会計（freee）と税務（TKC）をシームレスに統合
Claude Code運用とCowork/デスクトップ版運用の両対応を設計段階で組み込む
freeeを会計の基幹入力基盤、TKCを税額計算・申告基盤として位置付ける

# 2. office-claude設計原則（10原則）
これらの原則は、AI（Claude）と既存インフラ（freee/TKC）の役割分担を明確にし、長期的にメンテナンス可能な自動化基盤を構築するための憲法です。
原則1：freeeファースト（標準機能優先）
AI（Claude）でロジックを組む前に、freeeの標準機能（自動で経理、自動登録ルール、チェック機能等）で実現できないかを最優先で検討する。「AIはfreeeを操作する腕であり、freeeにある機能を再発明しない」ことを徹底する。
原則2：1 Skill = 1 判断 or 1 成果物
「このSkillが何をするか」を一文で簡潔に説明できる粒度を維持する。複雑な工程はSkillを分けることで、デバッグの容易性と再利用性を確保する。束ねるのはPluginの役割とする。
原則3：Skillの疎結合と単独実行
各Skillは独立して動作可能に設計する。複数のSkillを束ねる制御フローはPlugin層が担い、Skill自体に他者への依存を持たせない。
原則4：環境非依存（Multi-Platform Ready）
Claude Code（CUI）、Cowork、デスクトップ版のいずれのインターフェースからも呼び出し可能な共通設計とする。UIの差異は抽象化レイヤーで吸収する。
原則5：ロジックの外部参照化（Reference-Based）
判断基準や辞書、複雑な業務知識は references/ ディレクトリに集約する。SKILL.md 本体を軽量化することで、Claudeのコンテキスト消費を最小化し、ハルシネーションを抑制する。
原則6：出力形式のテンプレート化（Template-Driven）
レポートやメール、データ変換のフォーマットは templates/ を経由する。Skill本体に出力形式をハードコードせず、ビジネスルールの変更に柔軟に対応する。
原則7：計算とロジックの分離（Script-Isolated）
複雑な税務計算やデータ加工は scripts/（Python等）に切り出す。Skillから分離して単体テスト（Unit Test）を実行可能にすることで、計算精度の担保を行う。
原則8：標準Skillの最大活用
Excel/Word/PDF操作などは自前で実装せず、公式提供のSkillや標準機能を優先的に利用する。車輪の再発明を避け、メンテナンスコストを削減する。
原則9：ハイブリッド・命名規則
Claudeの認識精度（発火精度）を高めるため「日本語Skill名」を採用しつつ、システム管理の整合性のために「英語ファイル名」で構成する。
原則10：ヒューマン・イン・ザ・ループ（信任レベル管理）
すべてのSkillに「人確認レベル」を定義する。
低： 完全自動実行
中： 実行後に人間がログを確認（事後確認）
高： 実行前に人間の承認が必須（事前承認）

補足1：MCP接続は独立層で管理
認証情報・接続設定・接続確認を一元化し、保守性とテスト容易性を高める。

# 9. Claude Code Plugin対応方針
## 9.1 基本スタンス
office-claudeは「初期開発はスタンドアロン、後でプラグイン化」の二段階アプローチを採る。v1優先Skillの試行錯誤段階では .claude/ 配下で高速に開発し、複数スタッフへの配布やバージョン管理需要が明確になった段階で office-claude-plugin/ に移行する。

| 段階 | 構成 | タイミング |
| --- | --- | --- |
| Phase 1-2 | .claude/ 直下でスタンドアロン開発 | V1優先Skill実装期（1〜2ヶ月） |
| Phase 3- | office-claude-plugin/ にパッケージ化 | Cowork展開開始時 |

理由: プラグイン化の本質的価値（配布・バージョン管理・namespace隔離）は、複数スタッフへの展開段階で初めて必要になる。初期開発で導入するとnamespace付き呼び出しが試行錯誤の足枷になる。

## 9.2 スタンドアロン→プラグイン変換を容易にする5つの設計ルール

### ルール1: ディレクトリ構造をプラグイン互換に
ディレクトリ構造をプラグイン互換にする。7カテゴリ × 番号体系を .claude/skills/ 以下に物理化し、将来は skills/ へ機械的に移送できるようにする。
office-claude/
 ├── .claude/
 │   ├── skills/
 │   │   ├── verify/
 │   │   │   ├── V1-1-10-check-unprocessed-transactions/
 │   │   │   ├── V1-3-10-check-tax-classification/
 │   │   │   └── ...
 │   │   ├── intake/
 │   │   ├── classify/
 │   │   └── ...
 │   ├── agents/
 │   └── settings.json
 ├── data/		# 解析に使う素材（CSV, Excel, ログ等）
 ├── scripts/
 ├── references/	# 業務ルール、マニュアル、仕様書
 ├── reports/	# Claudeが作成したレポート（一時保存含む）
 ├── templates/	# レポート等に使うテンプレート
 ├── tests/		# プログラムの動作確認用
 │       └── e2e/       # E2Eテスト関連
 └── .mcp.json
7 ×  .claude/skills/  office-claude-plugin/skills/ 
### ルール2: パスはプロジェクトルート相対で書く
パスはプロジェクトルート相対で記述する。絶対パスやWindows固定パスは使わず、./scripts/foo.js のように統一する。
スクリプト呼び出しは ./scripts/foo.js のような相対パスで統一。絶対パスやハードコードされたWindowsパスを避ける。プラグイン化時に ${CLAUDE_PLUGIN_ROOT}/scripts/foo.js へ sed で一括変換可能にするため。
### ルール3: Skill技術名は英語kebab-case（原則9の徹底）
Skill技術名は英語 kebab-case とし、namespace付き呼び出しでも違和感のない命名にする。
namespace/office-claude:check-tax-classificationmodel-invoked
### ルール4: MCP設定を .mcp.json に独立（原則8の具体化）
MCP設定を .mcp.json に独立する。認証・起動・エンドポイントは接続層に閉じ込め、Skillには書かない。
後述の第10章で詳細。
### ルール5: 認証情報は .env で外出し
認証情報は .env で外出しする。将来 userConfig へ移行する際も構造を崩さない。
.mcp.json に直書きしない。将来 userConfig（プラグイン有効化時の対話入力）への移行時もスムーズ。

## 9.3 プラグイン化のトリガー条件
以下のいずれかが発生したらプラグイン化を開始する。
1.	Skill数が20を超える
2.	2人目以降のスタッフに展開する
3.	Cowork配布が具体化する

## 6.2 チェック観点
各フェーズでは、以下の6つの観点でチェックを行う（全観点が全フェーズで使われるわけではない）。

| 観点 | 定義 | 適用フェーズ |
| --- | --- | --- |
| 網羅性 | 漏れがないか | V1, V2, V3, V4, V5, V6 |
| 整合性 | 数値が合っているか | V1, V2, V3, V4, V5, V6 |
| ルール | 会計・税務ルールに従っているか | V1, V2, V3, V4, V5 |
| 異常 | 統計的・経験的に違和感がないか | V1, V3, V6 |
| 判断 | 専門的な会計・税務判断が必要か | V1, V3, V5 |
| 承認 | 責任者として最終Goを出せるか | V6のみ |

| 「判断」は担当者レベルの専門判断（会計処理・税務処理）。「承認」は責任者が提出のGoを出す行為。この2つを明確に分離することで、責任の所在が設計レベルで明確になる。 |

# 10. MCP接続層の独立管理（原則8の具体化）
## 10.1 設計方針
MCP接続は「接続層」と「判断層」を厳格に分離する。接続層はプロジェクトルート直下の .mcp.json で管理し、認証情報・エンドポイント・起動コマンドを集約する。業務Skillは、接続済みMCPツールを呼び出すだけの薄いラッパーとして実装する。

| レイヤー | 配置場所 | 役割 |
| --- | --- | --- |
| 接続層 | プロジェクトルート直下の .mcp.json | 認証情報・エンドポイント・起動コマンド |
| 業務Skill層 | .claude/skills/intake/ 配下 | 「freeeから総勘定元帳を取得する」等の業務判断 |
| 共通運用層 | .claude/skills/ops/ 配下 | 認証エラー、接続確認、再接続、監視、障害時ガイド |

業務Skillは .mcp.json で接続されたMCPツールを呼び出すだけの薄いラッパーとなる。Skill内にMCP接続ロジックを書かない。
## 10.2 .mcp.json の想定構成
初期は freee と Kintone を接続対象とし、Gmail・Chatwork・Drive は Phase 2 で追加する。TKCは公式MCP未提供を前提に、Phase 3 で CSV/PDF/Excel 経由の接続方式を検討する。

   "mcpServers": {
 	"freee": {
   	"command": "node",
   	"args": ["./scripts/mcp/freee-mcp-server.js"],
   	"env": {
     	"FREEE_CLIENT_ID": "${FREEE_CLIENT_ID}",
     	"FREEE_CLIENT_SECRET": "${FREEE_CLIENT_SECRET}",
     	"FREEE_REFRESH_TOKEN": "${FREEE_REFRESH_TOKEN}"
   	}
 	},
 	"kintone": {
   	"command": "npx",

     	"KINTONE_API_TOKEN_APP1": "${KINTONE_API_TOKEN_APP1}",

## 10.3 I1（intake / 収集）カテゴリとの関係
設計書v1.1の6.1節で定義されているI1のSkill群（freeeデータ取り込み、Gmail収集等）は、.mcp.json で接続されたMCPツールを呼び出す業務Skill として実装する。
例: I1-10-fetch-freee-general-ledger - SKILL.mdには「freee MCPの get_journals を呼び、指定期間・指定会社の総勘定元帳を取得する手順」を記述 - MCP接続の詳細は .mcp.json に閉じ込める - 認証エラー時のハンドリング等は共通化のためOカテゴリ（ops/環境管理）に切り出す。
v1.1で定義された I1 のSkill群は、.mcp.json で接続されたMCPツールを呼び出す業務Skillとして実装する。たとえば I1-10-fetch-freee-general-ledger は freee MCP の get_journals 相当機能を呼び出し、指定期間・指定会社の総勘定元帳を取得する。認証エラーや再接続判定は O3 に切り出す。
# 11. Agents層の活用（アーキテクト/実装者分離）

## 11.2 初期導入する4エージェント

| エージェント | モデル想定 | 主責務 | 備考 |
| --- | --- | --- | --- |

# 13. freee-autoプロジェクトからの資産引き継ぎ
## 13.1 基本方針
freee-auto は凍結・運用継続し、参照実装として活用する。office-claude への引き継ぎは、「MCP接続設定は .mcp.json として独立継承」「業務ロジックは移植マップを経由してSkill化」の二本柱で進める。
freee-auto（前身プロジェクト）は設計書v1.1の1.2節記載のとおり「凍結・運用継続、参照実装として活用」する。office-claudeへの引き継ぎは以下の方針で行う。
### 方針A: MCP接続設定 → .mcp.json 独立方式でそのまま引き継ぎ
freee-MCPとKintone-MCPはfreee-autoで既に動作実績がある。これらは Skill化せず、.mcp.json 接続設定として独立させて引き継ぐ。認証情報を .env で外出しし、環境変数経由で渡す。
この方式を選ぶ理由: - MCPは接続層であり、判断層（Skill）と混ぜない（原則8） - .mcp.json 単位で引き継げば、将来のプラグイン化時もファイル移動だけで済む - freee-auto側の変更と独立してoffice-claude側でバージョン管理できる
### 方針B: 業務ロジックは「移植マップ」を経由してSkill化
freee-autoの既存資産（normalizer.js、classifier系、66チェック項目、references/rules/、辞書類）は、いきなりSkill化せず、以下の2段階で引き継ぐ。
1.	移植マップ作成: freee-autoの各ファイル/モジュールが、office-claudeの7カテゴリ × 番号体系のどのSkillに該当するかの対応表を作る
2.	ドラフトSkill生成: マップを人間がレビューした後、Skillとして再構成する
この2段階にすることで、freee-autoの「積み上げ設計」のクセがoffice-claudeの粒度原則（原則1）に反映されないよう防げる。

| 方針 | 内容 | 理由 |
| --- | --- | --- |
| 方針A | freee-MCP / Kintone-MCP は .mcp.json として独立継承 | 接続層と判断層を混ぜない。プラグイン化時もファイル移動だけで済む。 |
| 方針B | normalizer.js、classifier系、66チェック項目、辞書類は「移植マップ」を作ってから再Skill化 | freee-autoの大粒度設計をそのまま持ち込まず、原則1に沿って分解する。 |

# 付録A 用語集

| 用語 | 定義 |
| --- | --- |
| Skill | 1つの判断または1つの成果物を生成する最小の業務単位 |
| Plugin | 複数のSkillを場面別に束ねたパッケージ（月次決算用、申告用等） |
| MCP | Model Context Protocol。外部システム（freee、Kintone等）との接続標準 |
| Subagent | 特定の専門的役割を持つAIエージェント（税務調査目線、帳簿レビュー目線等） |
| Cowork | Claudeデスクトップ版のファイル・タスク管理自動化ツール |
| freee-auto | 前身プロジェクト。凍結・運用継続中。参照実装として活用 |
| office-claude | 本プロジェクト。From Scratchで構築する新システム |
| 表示名 | Skillのカタログ上の名前（例：V1-3-10 消費税区分チェック） |
| 技術名 | SKILL.mdのnameフィールド（例：check-tax-classification） |
