# 1. 文書概要

## 1.1 目的

本仕様書は、office-claude のVerify層に属する Skill **`check-tax-classification`（V1-3-10）** の完成版仕様を定義する。
本Skillは、freee会計の総勘定元帳CSVを入力として受け取り、消費税区分の誤りまたは要確認の取引を検出して Finding 配列として出力する。

## 1.2 スコープ

- 日本の消費税法における基本的な税区分（課税/非課税/不課税/輸出免税/軽減税率）の判定誤りの検出
- 税法R01〜R12 ベースの安定領域の論点
- 売上側・仕入側の双方向検証

## 1.3 スコープ外

以下の論点は本Skillの責務外であり、別Skillで扱う：

| 論点                                                 | 担当Skill                            |
| ---------------------------------------------------- | ------------------------------------ |
| 軽減税率8%の精密判定                                 | V1-3-20 `check-reduced-tax-rate`     |
| 軽油引取税の按分判定                                 | V1-3-30（軽油引取税Skill、未着手）   |
| 海外サービス（電気通信利用役務）の判定               | V1-3-40 `check-overseas-service-tax` |
| 輸出免税・輸入消費税の判定                           | V1-3-50（輸出入消費税Skill、未着手） |
| インボイス制度（控80/控50経過措置等）                | 別Skill（未着手）                    |
| キーワード運用ベースのルール（諸会費・支払手数料等） | V1-3-90 `check-by-keyword-rules`     |
| 同一科目内の税区分ばらつき検知                       | `anomaly_detector`（既存）           |
| 通勤手当の課税仕入処理漏れ                           | V1-3-90（移管済み）                  |

## 1.4 関連Skillとの責務分離

- **V1-3-10 は税法ベースの安定領域**：条文・通達に根拠があり、変更頻度が低い論点を扱う
- **V1-3-90 は事務所ノウハウベース**：顧客対応で蓄積される独自ルール、頻繁に育つ領域を扱う
- 二つのSkillは番号体系上も離れて配置されており、性質の違いを明確化している

# 2. Skill全体仕様

## 2.1 Skill名

- 技術名：`check-tax-classification`
- 番号：V1-3-10
- 配置：`.claude/skills/verify/V1-3-rule/check-tax-classification/`

## 2.2 入力

- freee会計の総勘定元帳CSV（標準フォーマット）
- 仕訳1件単位で以下の情報を持つ：
  - 取引日（fiscal_year情報を含む）
  - 勘定科目名
  - 借方/貸方の判別
  - 借方金額・貸方金額
  - 税区分名
  - 取引先名
  - 摘要
  - 取引ID（walletTxnId等）

## 2.3 出力

Finding 配列。各 Finding は以下のフィールドを持つ：

```python
Finding = {
    "tc_code": str,           # "TC-01" 〜 "TC-08"
    "sub_code": str,          # "TC-01a" 等
    "severity": str,          # "🔴 High" / "🟡 Medium" / "🟢 Low"
    "area": str,              # "A1" 〜 "A13"
    "subarea": str,           # "rent", "land", "payroll" 等
    "error_type": str,        # 後述の4種類
    "show_by_default": bool,  # デフォルト表示するか
    "sort_priority": int,     # 出力時のソート優先度
    "wallet_txn_id": str,     # 取引ID
    "current_value": str,     # 現在の税区分
    "suggested_value": str,   # 推奨される税区分
    "confidence": int,        # 0〜100
    "message": str,           # ユーザー向け説明文(2.7のルールに従う)
    "deal_id": str,           # freee取引ID（freeeリンク生成に必須）
    "freee_general_ledger_url": str,  # 総勘定元帳URL（v1.2追加）
    "freee_journal_url": str,         # 仕訳帳URL（v1.2追加）
    "row_data": dict,         # 元の仕訳データ
}
```

## 2.4 読み取り専用原則

- 本Skillは検出のみを行い、freee会計の仕訳データを変更しない
- 自動修正機能は持たない
- 修正は税理士の最終判断のもと、人手または別ツールで実施する

## 2.5 severity 定義（office-claude 共通仕様）

**重要**：本severity定義は V1-3-10 単独の仕様ではなく、**office-claude 全Skillの共通仕様**である。
共通定義ファイル `references/common/severity-levels.md` で一元管理し、全SkillがそこからImportする。

### 2.5.1 共通仕様

| severity      | 意味                                                                 |
| ------------- | -------------------------------------------------------------------- |
| **🔴 High**   | 明確な誤り・税務リスク・法令違反の可能性。即対応が必要。             |
| **🟡 Medium** | 確認が必要。文脈次第で問題ない場合もあるが、税理士の目視確認を推奨。 |
| **🟢 Low**    | 情報提供・参考値。対応は任意。                                       |

### 2.5.2 設計の意図

- **信号機メタファー**：🔴→🟡→🟢 で直感的に緊急度が伝わる
- **英語併記**：High/Medium/Low により、Excel集計・Kintone連携・スタッフ間の共通言語化が容易
- **緊急度ベース**：「要修正/要確認/情報」のような行動指示ではなく、**緊急度の段階**として明確化

### 2.5.3 freee-auto との関係

- freee-auto側（既存）の `🔴 要修正 / 🟡 要確認 / 🔵 情報` はそのまま動かし続ける
- office-claude の新Skillは本仕様（🔴 High / 🟡 Medium / 🟢 Low）で統一する
- 将来的に freee-auto を office-claude に統合する際は、変換マッピングを別途用意する

## 2.6 error_type 定義

`error_type` は Finding の性質を4階層で表現する：

| error_type        | 性質                                                                 | 典型的な severity   | 表示                      |
| ----------------- | -------------------------------------------------------------------- | ------------------- | ------------------------- |
| `direct_error`    | 確定誤り。機械判定で「ほぼ間違いない」と言えるもの                   | 🔴 High / 🟡 Medium | デフォルト表示            |
| `mild_warning`    | 軽微な精度向上。税額影響は軽微だが帳簿の正確性向上のため             | 🟢 Low              | `show_by_default = false` |
| `gray_review`     | グレー論点。機械判定では決められず人確認必須                         | 🟡 Medium           | デフォルト表示            |
| `reverse_suspect` | 逆方向疑義。「正解寄り」の処理になっているが摘要に逆方向の特徴がある | 🟡 Medium           | デフォルト表示            |

## 2.7 message 設計ルール

Finding の `message` フィールドは、以下の3文構造で記述する：

```
[1文目] 結論：何が誤り（または要確認）か
[2文目] 理由：税法ベースの根拠
[3文目] 対応：何を確認すべきか
```

### 2.7.1 例

#### TC-03a（給与系が課税仕入）

```
給与・賞与・役員報酬等は対象外です。
労働の対価は資産の譲渡等に該当しないため、消費税の対象外です。
科目「給与手当」が課税仕入となっているため、対象外への変更を確認してください。
```

#### TC-02c（住宅家賃が課税仕入）

```
住宅用家賃の支払は非課税です。
消費税法 別表第一 第13号により、住宅の貸付けは非課税取引とされています。
摘要に「社宅」「寮」等の住宅用キーワードがあるため、内容を確認してください。
```

### 2.7.2 例外

- 短文で十分な情報量がある場合は1〜2文に簡略化してよい
- 逆に複雑なグレー論点では4〜5文に拡張してもよい
- 重要なのは「結論→理由→対応」の論理構造を保つこと

### 2.7.3 トーン

- **断定しすぎない**：「必ず」「絶対」よりも「〜の可能性があります」「〜を確認してください」
- **専門用語の説明は最小限**：税理士向けのため、基本用語の解説は不要
- **アクションオリエンテッド**：何をすべきかが明確になる文末

## 2.8 confidence 基準

| confidence | 意味           | 典型的な検出条件                                                 |
| ---------- | -------------- | ---------------------------------------------------------------- |
| **90以上** | 科目だけで確定 | 「給与手当」が課対仕入になっている等、科目だけで誤りが断定できる |
| **80〜89** | 科目 + 強いKW  | 「地代家賃 × 課対仕入 × 社宅」のように、科目と強い住宅KWが揃う   |
| **70〜79** | 科目 + 弱いKW  | 「地代家賃 × 課対仕入 × マンション」のように弱いKWで補強         |
| **60〜69** | KWベース推測   | 摘要キーワードのみで判定するgray_review系                        |
| **50以下** | グレー         | 複数の解釈が成立し得る、人確認必須レベル                         |

### 2.8.1 設計の意図

- 横断基準を置くことで、TC間でのconfidence値の整合性が取れる
- 将来のAIチューニング時に、confidenceを学習データのラベルとして使える
- ユーザーが「confidence 70以上だけ表示」のようなフィルタ運用が可能

## 2.9 show_by_default の考え方

- `show_by_default = true`：レポートのメイン領域に表示
- `show_by_default = false`：折りたたみ領域 or 参考情報セクションに表示
- 主な対象：`mild_warning`（TC-XXc 系）、別Skill本命の気づきポイント（TC-06d, TC-07f）

## 2.10 sort_priority

- 出力時のソート優先度（**表示順の制御軸**）
- 数値が小さいほど優先度が高い
- **severity とは切り離して独立した軸として扱う**
- 同じ severity でも、表示順を細かく制御するために使う

### 2.10.1 標準レンジ（v1.2.1 修正版）

| レンジ     | 用途           | 説明                                            |
| ---------- | -------------- | ----------------------------------------------- |
| **1〜10**  | 最優先レビュー | 最初に確認すべき重大な誤り                      |
| **11〜30** | 通常レビュー   | 通常の監査フローで確認する論点                  |
| **31〜50** | 軽微・参考     | 影響軽微だが帳簿の正確性向上のための指摘        |
| **90台**   | 初期非表示     | mild_warning 等、フィルタで初期非表示にする項目 |

### 2.10.2 severity との関係

- sort_priority は severity と**独立した軸**である
- 例えば 🔴 High でも sort=11 のケースがあり得るし、🟡 Medium でも sort=5 のケースがあり得る
- ただし、実務上の傾向としては：
  - 1〜10：🔴 High が中心、ただし🟡 Mediumの中でも特に重要なものを含む
  - 11〜30：🟡 Medium が中心、🔴 Highの一部も含む
  - 31〜50：🟢 Low が中心
  - 90台：mild_warning（🟢 Low、show_by_default = false）

### 2.10.3 設計の意図

- レンジを先に固定することで、Excel出力時のソート設計が安定する
- severity と sort_priority を切り離すことで、同じ severity 内での細かい優先制御が可能
- 将来TC追加時も、このレンジ内に収めれば全体ソート順が崩れない
- 90台を初期非表示専用レンジとすることで、フィルタ設計が単純化される

## TC-01：売上の税区分検証

### 5.1.1 観点の定義

**目的**：売上科目について、現在ついている税区分が摘要・取引先の内容と整合しているかを検証し、誤りまたは要確認の取引を検出する。

### 5.2.4 住宅判定の3層ロジック

```python
def classify_residential(desc, keywords):
    has_strong = matches_any(desc, keywords.residential_strong)
    has_weak = matches_any(desc, keywords.residential_weak)
    has_business = matches_any(desc, keywords.business_use)

    if has_strong and has_business:
        return "mixed"  # 両方あり→要確認
    if has_strong:
        return "strong"
    if has_weak and has_business:
        return "business_override"  # 事業用優先
    if has_weak:
        return "weak"
    return "none"
```

## TC-05：支払利息・保険料の非課税漏れ

## TC-07：福利厚生費の不課税・非課税判定誤り

# 12. freeeリンク仕様（STEP 4-C 重要追加要件）

## 12.1 設計思想

レポートでフラグが立った内容は、最終的に**必ず freee に立ち返って確認する運用**になる。
そのため、Excel の Finding 各行から **freee へ直接飛べるリンクを必須要件**とする。
これにより Excel が単なる出力物ではなく、**freee 確認の起点となるレビュー画面**として機能する。

## 12.2 リンクの2種類

### 12.2.1 タイプ1：総勘定元帳リンク

**用途**：同じ科目・同じ期間・必要に応じて税区分込みで**俯瞰確認**するためのリンク

**URL構成例（実装時に実測ベースで確定）**：

```
https://accounting.secure.freee.co.jp/reports/general_ledger
  ?company_id={company_id}
  &fiscal_year_id={fiscal_year_id}
  &account_item_id={account_item_id}
  &start_date={start_date}
  &end_date={end_date}
  &tax_group_codes={tax_group_codes}
```

**生成に必要なパラメータ**：

| パラメータ                | 取得元         | 備考                                |
| ------------------------- | -------------- | ----------------------------------- |
| `company_id`              | 事業所マスタ   | 事務所別の固定値                    |
| `fiscal_year_id`          | 会計期マスタ   | 期首年から決定                      |
| `account_item_id`         | 科目マスタ     | 科目名 → ID変換が必要               |
| `start_date` / `end_date` | 取引日から計算 | **月単位**（取引日の月の1日〜末日） |
| `tax_group_codes`         | 税区分マスタ   | 任意（絞り込み用）                  |

**期間指定の方針**：

- **詳細シート**：取引日の**月単位**（例：2026/03/01〜2026/03/31）
- **サマリーシート**：将来は**期単位**（年度全体）も持たせる余地あり

### 12.2.2 タイプ2：仕訳帳リンク（deal_id ベース）

**用途**：該当の deal_id をピンポイントで確認

**URL構成例（実装時に実測ベースで確定）**：

```
https://accounting.secure.freee.co.jp/deals/{deal_id}
```

**生成に必要なパラメータ**：

| パラメータ | 取得元               | 備考 |
| ---------- | -------------------- | ---- |
| `deal_id`  | 仕訳データに含まれる | 必須 |

## 12.3 URL仕様の確定方針（重要）

本仕様書に記載した URL パターンはあくまで参考形式である。
**実装時には、悠皓さんが実際の freee 画面から確認した URL パターンを正とし、推測ベースの実装を避ける**。
パラメータ名・順序・必須項目はすべて実測で確定する。
