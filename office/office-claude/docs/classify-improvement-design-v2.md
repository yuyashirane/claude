## 1. 改善後のフロー（全体像）

```
[入力]
  未処理明細 + 既存ルールCSV（任意） + 過去取引データ

    ↓

[Phase A] 取引類型判定
  ① 借入返済（ｺﾞﾍﾝｻｲ等）→ 複合仕訳推測
  ② ATM引出・預入（ｼﾞﾄﾞｳｷ等）→ 振替推測
  ③ クレカ引落（銀行口座からｸﾚｼﾞﾂﾄ）→ 振替推測
  ④ 口座間振替（自社口座間）→ 振替推測
  ⑤ 売上入金（入金側パターン）→ 売上推測
  ⑥ 個人宛支払（人名パターン）→ 中間類型として保持★
  ⑦ 通常経費支出（上記以外）→ Phase Bへ

    ↓

[Phase B] 多段階推測（勘定科目・税区分ともに同じ優先順）★
  ① 既存自動登録ルール照合（条件全体を照合）★
  ② freee過去取引パターン照合
  ③ 顧問先固有辞書
  ④ 業種辞書
  ⑤ 一般キーワード辞書（22科目）
  ⑥ 未判定（「要確認」フラグ、一律雑費にしない）★

    ↓

[Phase C] 取引先・品目・マッチ条件の生成
  - 引落代行表現の除去 + カタカナ正規化
  - 取引先タグルール準拠の名称生成
  - 品目タグマッピング
  - 部分一致優先のマッチ条件判定

    ↓

[Phase D] レビュー用Excel出力（修正書き込み可能な形式）★
  - 推測結果列 + 修正用空列 + 確定フラグ

    ↓

[ユーザー確認・修正（Excelに直接書き込み）]

    ↓

[Phase E] 修正済みExcelを読み込み → 最終CSV生成 + 顧問先固有辞書更新★
```

★印: v1からの変更・追加点

## 2. v1からの変更点（悠皓さんフィードバック6点の反映）

### 変更1: 税区分も多段階推測の対象

勘定科目と同じ6段階の優先順で推測する。
不明な場合は「課対仕入」に一律化せず、「要確認」として出力。

```javascript
// Phase Bの戻り値構造
{
  account: '通信費',          // 勘定科目の推測結果
  accountSource: 'existing_rule',
  accountConfidence: 90,
  
  taxClass: '課対仕入',       // 税区分の推測結果 ★独立して推測
  taxClassSource: 'existing_rule',
  taxClassConfidence: 90,
  
  // 税区分が不明な場合:
  // taxClass: null,
  // taxClassSource: 'unmatched',
  // taxClassConfidence: 0,
  // → レビュー用Excelで「要確認」ハイライト
}
```

### 変更2: 人名パターン → 中間類型「個人宛支払」

Phase Aで人名パターンを検出した場合、`PERSONAL_PAYMENT`（個人宛支払）として
中間類型に分類する。直ちに「給料手当」に確定しない。

```javascript
// Phase A の判定
if (isPersonName(desc)) {
  return {
    type: 'PERSONAL_PAYMENT',  // ★「給料手当」ではなく中間類型
    confidence: 60,
    note: '個人宛支払。給与/外注費/立替精算/役員関連等の可能性あり',
  };
}

// Phase Bで最終推測
// - 既存ルールに「給料手当」の同名取引があれば → 給料手当
// - 過去取引で同名の外注費取引があれば → 外注費
// - 金額帯が毎月固定 → 給料手当の可能性高
// - 不定期/不定額 → 外注費 or 立替の可能性
// - 上記で判定できない → 「要確認」（個人宛支払）
```

### 変更3: 既存ルール照合は条件全体を照合

取引内容とマッチ条件だけでなく、7項目を照合対象にする。

```javascript
function matchExistingRules(item, existingRules) {
  for (const rule of existingRules) {
    // 1. 取引内容のマッチ（完全一致/部分一致/前方一致）
    if (!matchText(item.description, rule['取引内容'], rule['マッチ条件'])) continue;
    
    // 2. 収支区分の一致
    if (rule['収支区分'] && rule['収支区分'] !== item.entrySideJa) continue;
    
    // 3. 取引口座の一致（指定ありの場合のみ）
    if (rule['取引口座'] && rule['取引口座'] !== item.walletableName) continue;
    
    // 4. 金額条件（最小値・最大値）
    if (rule['金額（最小値）'] && item.amount < Number(rule['金額（最小値）'])) continue;
    if (rule['金額（最大値）'] && item.amount > Number(rule['金額（最大値）'])) continue;
    
    // 全条件一致 → ルールの内容を採用
    return {
      account: rule['勘定科目'],
      taxClass: rule['税区分'],
      partner: rule['取引先'],
      item: rule['品目'],
      action: rule['マッチ後のアクション'],
      transferAccount: rule['振替口座'],
      confidence: 92,   // 条件全体一致は高信頼度
      source: 'existing_rule',
      matchedRule: rule['取引内容'],
      note: `既存ルール一致（優先度: ${rule['優先度'] || 0}）`,
    };
  }
  return null;
}
```

### 変更4: レビュー用Excelに修正用列を追加

推測結果列に加えて、修正を直接書き込める空列を配置。

| 列グループ | 列名 | 用途 |
|-----------|------|------|
| 基本情報 | 元明細 / 口座 / 金額 / 入出金 | 元データ（編集不可） |
| 推測結果 | 取引類型 / 勘定科目 / 税区分 / 取引先 / 品目 / マッチ条件 / マッチテキスト / アクション | AI推測（参考） |
| 推測根拠 | 推測ソース / 推測理由 / 信頼度 | 判断材料 |
| **修正用** | **修正後科目 / 修正後税区分 / 修正後取引先 / 修正後品目 / 修正後マッチ条件 / 修正後マッチテキスト / 修正後アクション / 修正コメント / 確定フラグ** | **★ユーザーが書き込む** |

- 修正用列は薄い背景色で「ここに書き込んでください」と分かる見た目にする
- 確定フラグ: ✅（確定）/ 空欄（未確認）/ ❌（除外）
- 修正用列が空の場合は推測結果をそのまま採用

### 変更5: 初回は安全側で振り分け

フェーズ1では自動登録候補を広げず、以下の方針:

| 信頼度 | 初回フェーズ1の扱い | 将来（信頼確立後） |
|--------|-------------------|------------------|
| 85-100 | **推測する**（自動登録にしない）★ | 取引を登録する |
| 45-84 | **推測する** | 取引を推測する |
| 1-44 | **要確認** | Kintone送り |
| 0 | **要確認** | Kintone送り |

つまりフェーズ1では「取引を登録する」は使わない。
全件が「取引を推測する」か「要確認」のいずれかになる。
これは初回レビューで品質を確認してから自動登録範囲を広げるため。

## 4. フェーズ1完了後の判断ポイント

レビュー用Excelを悠皓さんに確認いただき、以下を判断:

1. **取引類型の切り分けは妥当か** — 借入返済・ATM・クレカ引落が正しく分類されているか
2. **勘定科目の推測精度** — 雑費率が大幅に減っているか、推測が実務感に合っているか
3. **税区分の推測精度** — 不明なものが「要確認」になっているか
4. **取引先タグの正規化** — 引落代行表現が除去され、法人格略称が適用されているか
5. **レビュー形式** — 修正を書き込みやすいか、確定フラグが使いやすいか

上記OKであれば、フェーズ2（過去取引照合・業種辞書・品目マッピング）に進む。
修正が必要であれば、フィードバックを反映してフェーズ1を再実行。
